Market & Competitor Analysis
Tanzania's digital media market is one of East Africa's fastest-growing, driven by rapid smartphone penetration,
affordable mobile data from Vodacom, Airtel, and Halotel, and a predominantly young population eager for local
content. Yet the ecosystem remains underserved: legacy news sites built in the mid-2000s dominate, offering poor
mobile UX, slow load times, and minimal personalization. Farols enters this space with a modern, performance-first
architecture positioned to capture both urban tech-savvy readers and semi-urban mobile-first audiences.

1.2 Content Strategy Patterns
 Breaking News Aggregation: Major outlets repurpose wire content (AFP, Reuters) with minimal local
context. First-mover advantage is rarely exploited due to slow CMS workflows.
 Swahili vs. English split: ~65% of content is in Swahili but most tech coverage is English-only — a
significant gap for localization.
 Mobile reading patterns: Average session length is under 90 seconds; users skim headlines and share to
WhatsApp groups rather than commenting on-site.
 Comment culture: On-site commenting is nearly extinct. Facebook and WhatsApp Groups are the de-facto
comment sections for Tanzanian readers.
 Monetization maturity: Most platforms rely purely on Google AdSense. Programmatic advertising, native
ads, and newsletter monetization are virtually unused.
 Social sharing: WhatsApp is the #1 sharing channel (estimated 70% of shares), followed by Facebook, then
Twitter/X. Telegram groups are emerging.

1.4 Market Gaps Farols Can Exploit
 Tech + Breaking News hybrid: No platform combines fast-breaking news with deep tech analysis in a single
bilingual product.

SECTION 02
Product Vision
Farols is positioned as a modern, performant, socially-integrated content platform that serves Tanzanian
readers who want fast, trustworthy tech and news coverage — accessible on a 3G connection, readable in both
English and Swahili, and shareable to WhatsApp in one tap. The product philosophy borrows from Medium's
editorial depth, Twitter's real-time speed, and Reddit's community engagement — rebuilt from the ground up for the
East African mobile context.
2.1 Core Value Proposition
For Tanzanian readers aged 18–40: Farols is the only platform that delivers breaking tech and news content that
loads in under 2 seconds on mobile, personalizes your feed based on what you read, and lets you share directly to
WhatsApp — in English or Swahili.

2.3 Differentiation Strategy
 Speed as a feature: Lighthouse score ≥90 mobile is a product requirement, not a nice-to-have. Every release
is gated on performance budgets.
 Bilingual by design: The CMS supports parallel English/Swahili fields. The front-end switches language
without a page reload.
 WhatsApp-first sharing: Share metadata (OG image + title) is auto-generated per article. WhatsApp
previews are tested before every publish.
 Real-time breaking news: A dedicated fast-publish mode bypasses the editorial queue for breaking news,
pushing to all subscribers via WebSocket within 30 seconds of publish.
 Engagement without toxicity: Threaded comments with upvote/downvote, ML-assisted spam filter, and
one-click moderation reduce the noise that drove Tanzanian readers to WhatsApp groups.
 Writer-centric economics: Writer dashboards with read counts, engagement heatmaps, and a future tip-jar
feature build platform loyalty among content creators.

SECTION 03
System Architecture
The Farols stack is deliberately chosen for a lean team: React on the frontend for component reuse and ecosystem
richness; Express.js on the backend for simplicity and Node.js alignment; SQLite for zero-ops local storage during
MVP with a clean migration path to PostgreSQL. The architecture is service-boundary-aware so individual
components can be extracted into microservices as traffic grows.

Routing Strategy
React Router v6 with lazy-loaded routes. Every route component is wrapped in React.lazy() and Suspense with a

routes additionally check the user's role claim from the JWT payload.

SEO Strategy
 Server-side rendering: For MVP, use react-snap or a lightweight SSR via Express to pre-render article
pages for Googlebot. Full Next.js migration is planned for Phase 3.
 React Helmet Async: Every page sets , meta description, og:image, og:title, og:type, article:published_time,
article:author dynamically.
 Structured data: Article schema (JSON-LD) injected into every article page for Google News eligibility.
 Sitemap: Auto-generated sitemap.xml from Express route that queries all published post slugs. Submitted to
Google Search Console.
 Canonical URLs: Every article has a canonical link element to prevent duplicate content penalties from
category/tag URL variants.
Performance Optimization
 Code splitting: Vite's automatic chunk splitting per route. Vendor chunk separated from app code.
 Image optimization: All user-uploaded images auto-converted to WebP via Sharp on the backend.
Responsive img tags with srcSet generated for 400w, 800w, 1200w.
 Lazy loading: Images below the fold use loading='lazy'. IntersectionObserver triggers article list pagination.
 Font subsetting: Google Fonts loaded with display=swap and preloaded. Only latin and latin-ext subsets to
minimize KB.
 Service Worker (PWA): Workbox precaches shell (HTML, CSS, JS). Runtime caches article pages with
stale-while-revalidate strategy for 24h.
 Bundle budget: Main bundle < 150KB gzipped. Lighthouse mobile score target ≥ 90.

Request →
  1. helmet()              Security headers (CSP, X-Frame-Options, etc.)

  9. errorHandler()        Catch all throws, format JSON error
 10. Response
Authentication — JWT + Refresh Token
 Access token: HS256 JWT, 15-minute expiry. Payload: {userId, username, role, iat, exp}.
 Refresh token: 64-byte random hex stored in DB (refresh_tokens table). 30-day expiry. Sent as httpOnly
cookie; not accessible to JS.
 Token rotation: Each refresh call issues a new access token AND a new refresh token (old one deleted).
Detects reuse attacks.
 Password hashing: bcrypt with cost factor 12. Minimum password 8 chars enforced at validation layer.

3.3 Database — SQLite (better-sqlite3)
SQLite in WAL (Write-Ahead Logging) mode supports concurrent reads and sequential writes suitable for an
early-stage platform handling up to ~1,000 concurrent users. All queries use parameterized statements (never
string concatenation). The schema is normalized to 3NF. FTS5 virtual tables power full-text search.
Pragmas (applied on connection init)
PRAGMA journal_mode = WAL;       -- concurrent reads
PRAGMA synchronous = NORMAL;     -- balance durability/speed
PRAGMA foreign_keys = ON;        -- enforce FK constraints
PRAGMA cache_size = -64000;      -- 64MB page cache
PRAGMA temp_store = MEMORY;      -- temp tables in RAM
PRAGMA mmap_size = 268435456;    -- 256MB memory-mapped IO

4.1 Authentication & User System
Registration Flow
1. POST /api/v1/auth/register
   Body: { email, username, password, displayName }
   → Validate: email format, username /^[a-z0-9_]{3,30}$/, password ≥8 chars

4.4 Engagement Features
Likes System
The likes system uses a toggle model (like = insert, unlike = delete from likes table) with optimistic UI updates to
mask network latency — critical on slow Tanzanian connections.
API:  POST /api/v1/posts/:id/like  (auth required)
Server logic:
  const existing = db.prepare('SELECT 1 FROM likes WHERE user_id=? AND post_id=?')
                     .get(userId, postId);
  if (existing) {
    db.prepare('DELETE FROM likes WHERE user_id=? AND post_id=?').run(userId, postId);
    return { liked: false, count: getLikeCount(postId) };
  } else {
    db.prepare('INSERT INTO likes(user_id,post_id) VALUES(?,?)').run(userId, postId);
    createNotification(post.author_id, 'like', userId, postId);
    return { liked: true, count: getLikeCount(postId) };
  }
Rate limit: 10 likes/minute per user (prevent bot inflation).
Optimistic UI (React Query mutation):
  onMutate: update local cache immediately (flip liked bool, ±1 count)
  onError:  roll back to previous cache state
  onSettled: refetch to sync truth from server
Comments System
Threaded comments support two levels: root comments and replies. Deeper nesting is intentionally avoided to
prevent conversation fragmentation common on platforms like Reddit. The tree is built client-side from a flat list
returned by the API (parent_id grouping).
Fetch comments:
  GET /api/v1/posts/:id/comments?page=1&limit;=20&sort;=top|new
  Query (top sort):
    SELECT c.*, u.username, u.avatar_url,
           (SELECT vote FROM comment_votes WHERE user_id=:me AND comment_id=c.id) AS my_vote
    FROM comments c JOIN users u ON c.author_id=u.id
    WHERE c.post_id=:postId AND c.parent_id IS NULL AND c.is_removed=0
    ORDER BY c.score DESC, c.created_at DESC
    LIMIT :limit OFFSET :offset
Fetch replies for a root comment:
  GET /api/v1/comments/:id/replies
  Same query with WHERE parent_id=:id
Upvote/Downvote:
  PUT /api/v1/comments/:id/vote  Body: { vote: 1 | -1 | 0 }
  → INSERT OR REPLACE INTO comment_votes
  → UPDATE comments SET score = score + delta
  → delta = newVote - (oldVote ?? 0)
Moderation:
  PUT /api/v1/comments/:id/flag     — user flags comment
  DELETE /api/v1/comments/:id       — removes own comment (sets is_removed=1)
  Admin: GET /api/v1/admin/flagged-comments  — moderation queue
         PUT /api/v1/admin/comments/:id/remove  — admin removal
Spam filter (Phase 2): Akismet API or a local keyword blocklist on insert.

Trending Algorithm
Trending score is calculated on-the-fly, not stored, to avoid stale ranks. The algorithm is a decay function weighted
by engagement quality:
trending_score = (
  (likes * 3) +
  (comments * 5) +
  (shares * 4) +
  (views * 0.5)
) / POWER((hours_since_published + 2), 1.5)
-- hours_since_published = (unixepoch('now') - unixepoch(published_at)) / 3600.0
-- Gravity factor 1.5 decays older content faster than Hacker News (1.8)
-- Breaking news posts get +500 flat bonus for first 2 hours
Cached in memory (Map) and refreshed every 5 minutes by a
node-cron job to avoid re-computing on every feed request.
Personalized Feed Algorithm
Signals from user_reads table:
  - category affinity:  count reads per category → weight 0.0–1.0
  - tag affinity:       count reads per tag via post_tags join
  - author affinity:    count reads per author (future: explicit follows)
  - recency of reads:   exponential decay, reads >7 days old weighted 0.1
Feed query (simplified):
  SELECT p.*,
    (cat_weight * 0.4 + tag_weight * 0.3 + author_weight * 0.2
     + trending_score_normalized * 0.1) AS personalization_score
  FROM posts p
  WHERE p.status = 'published'
    AND p.id NOT IN (SELECT post_id FROM user_reads WHERE user_id = :userId)
  ORDER BY personalization_score DESC
  LIMIT 20 OFFSET :offset
Cold start (new user, no read history):
  Fall back to trending feed for first 10 articles.
  After 10 reads, personalization kicks in.

Performance & Scalability
6.1 SQLite Scaling Strategy
SQLite with WAL mode handles approximately 1,000 concurrent read connections and ~100 writes/second on a
modern VPS — sufficient for Farols' first 50,000 monthly active users. The migration path to PostgreSQL is
designed to be low-friction:
Phase 1 (0–50K MAU):   SQLite + WAL + better-sqlite3
Phase 2 (50K–200K):    PostgreSQL on same VPS (schema is ANSI SQL-compatible)
  Migration: export SQLite → pg_restore via pgloader (handles type mapping)
  Zero-downtime: run both DBs in parallel, migrate reads first, then writes
Phase 3 (200K+):       PostgreSQL + read replicas (PgPool-II or AWS RDS Multi-AZ)
  Add Redis for session cache, rate-limit counters, trending score cache
  Move file uploads to S3-compatible object storage (Cloudflare R2)
